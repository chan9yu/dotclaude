# Frontend Fundamentals — 변경하기 쉬운 코드

## 이 스킬이 추구하는 단 하나의 기준

좋은 프론트엔드 코드는 **변경하기 쉬운 코드**입니다. 새 요구사항이 왔을 때 기존 코드를 수정하고 배포하기 수월하면 좋은 코드입니다. "변경하기 쉬운가?"를 4가지 렌즈로 진단합니다.

| 렌즈 | 질문 | 낮을 때의 증상 |
|------|------|----------------|
| **가독성** (Readability) | 읽는 사람이 한 번에 떠올려야 하는 맥락이 적은가? | 한 함수에 분기·맥락이 뒤엉켜 동작을 한눈에 못 읽음 |
| **예측 가능성** (Predictability) | 이름·파라미터·반환값만 보고 동작을 맞힐 수 있는가? | 같은 이름이 다른 동작, 숨은 부수효과, 들쭉날쭉한 반환 타입 |
| **응집도** (Cohesion) | 함께 고쳐야 할 코드가 반드시 함께 고쳐지는 구조인가? | 한쪽만 고치면 조용히 깨짐(매직 넘버, 흩어진 파일) |
| **결합도** (Coupling) | 한 곳을 고쳤을 때 영향 범위가 좁고 예측 가능한가? | 공통 코드 하나 고치면 수십 곳이 흔들림, props drilling |

> 핵심 통찰: 사람이 한 번에 머릿속에 담는 맥락은 6~7개로 제한됩니다. 가독성 전략의 대부분은 "이 줄을 읽는 사람이 동시에 떠올려야 하는 것의 개수를 줄이기"로 환원됩니다.

## ⚖️ 가장 중요한 것: 4가지는 서로 상충한다

**이 스킬의 정수는 규칙 적용이 아니라 트레이드오프 판단입니다.** 4가지를 동시에 최대화할 수 없습니다. 규칙을 기계적으로 적용하지 말고, 어떤 가치를 우선할지 *상황에 근거해* 결정하고 그 근거를 밝히세요.

- **가독성 ↔ 응집도**: 함께 수정되도록 공통화·추상화하면 응집도는 오르지만, 한 겹 추상화되어 가독성은 떨어집니다.
  - → 함께 안 고치면 **버그가 나는** 경우엔 응집도 우선(공통화·추상화). 위험이 낮으면 가독성 우선(중복 허용).
- **응집도 ↔ 결합도**: 중복을 제거해 하나로 묶으면 응집도는 오르지만 결합도도 같이 올라(영향 범위 확대) 수정이 어려워질 수 있습니다.
  - → 페이지마다 동작이 **달라질 여지**가 있으면 중복을 허용해 결합도를 낮추세요. 동작이 확실히 같고 앞으로도 같을 때만 공통화하세요.

추상화·공통화는 "지금 읽기 좋아 보이는가"가 아니라 **"미래에 함께 바뀔 운명인가"**로 결정합니다. 섣부른 공통화는 가장 흔한 실수입니다.

## 진단 → 개선 워크플로우

코드를 받으면(또는 작성·리뷰할 때) 이 순서로 진행하세요. 문서 원본의 `코드 예시 → 👃 코드 냄새 맡기 → ✏️ 개선하기` 흐름을 그대로 따릅니다.

1. **냄새 맡기 (진단)** — 아래 "빠른 냄새 감지 인덱스"로 코드를 훑어 어떤 기준이 깨졌는지 *이름을 붙여* 식별합니다. 해당 패턴의 references 파일을 읽어 정확한 근거와 개선 기법을 확인하세요.
2. **트레이드오프 점검** — 개선이 다른 축을 해치지 않는지 위 ⚖️ 섹션으로 확인합니다. 특히 "추상화/공통화하라"는 제안 전에는 *반드시* 미래 변경 가능성을 따져보세요.
3. **개선안 제시** — before/after를 짝지어 보여주고, *왜* 더 변경하기 쉬워지는지(어떤 맥락이 줄고 어떤 영향 범위가 좁아지는지)를 설명합니다. 한 번에 모든 걸 바꾸지 말고 영향 큰 것부터.
4. **판단 보류 명시** — 정보가 부족해 단정할 수 없으면(예: 이 중복이 미래에 갈라질지 모름) 그 점을 드러내고 질문하세요. 단정적 규칙 강요보다 맥락 확인이 낫습니다.

### 진단 리포트 형식

리뷰/진단을 요청받으면 이 구조로 출력하세요:

```
## 코드 냄새 진단

### 🔴 [기준] 한 줄 요약  (예: 결합도 — usePageState가 모든 책임을 떠안음)
- **무엇이**: 문제 위치와 현상
- **왜 변경하기 어려운가**: 어떤 맥락/영향 범위가 문제인지
- **개선**: before → after (핵심만), 적용할 패턴 이름

### ⚖️ 트레이드오프 메모 (해당 시)
- 이 개선이 다른 축에 주는 영향, 우선순위 판단 근거
```

심각도는 "변경하기 어려운 정도"로 매기세요: 🔴 지금 버그·광범위 영향을 부르는 구조 / 🟡 이해를 더디게 하는 가독성 문제 / 🟢 취향·소규모 개선.

## 빠른 냄새 감지 인덱스

코드에서 아래 신호가 보이면 괄호 안 패턴을 의심하고 해당 references를 읽으세요.

### 가독성 → `references/readability.md`
- 한 컴포넌트/함수가 **동시에 실행되지 않는 분기**(viewer vs admin 등)를 뒤섞어 처리 → *같이 실행되지 않는 코드 분리하기*
- 로그인 체크·권한 가드 등 **구현 상세가 본문에 그대로 노출** → *구현 상세 추상화하기* (Wrapper/HOC)
- 쿼리·상태·API를 **종류별로 한 Hook에 몰아넣음**(`usePageState`류) → *로직 종류로 합쳐진 함수 쪼개기*
- `filter`/`some`/`&&`가 여러 겹 중첩된 **이름 없는 복잡한 조건** → *복잡한 조건에 이름 붙이기*
- 의미를 알 수 없는 **숫자 리터럴**(`delay(300)`) → *매직 넘버에 이름 붙이기*
- 이유를 알려고 **위아래/여러 함수를 오가야 함**(`POLICY_SET`류 과한 추상화) → *시점 이동 줄이기*
- **중첩 삼항 연산자** → *삼항 연산자 단순하게 하기*
- 범위 비교가 `a >= b && a <= c`처럼 **중간값을 두 번** 확인 → *왼쪽에서 오른쪽으로 읽히게 하기*

### 예측 가능성 → `references/predictability.md`
- 라이브러리와 **같은 이름**인데 동작이 다른 래퍼(`http.get`이 토큰까지 처리) → *이름 겹치지 않게 관리하기*
- 같은 종류 함수가 **제각각 반환 타입**(한쪽은 Query, 한쪽은 data) → *반환 타입 통일하기*
- 이름·반환값에 안 드러나는 **숨은 부수효과**(`fetchBalance`가 몰래 로깅) → *숨은 로직 드러내기*

### 응집도 → `references/cohesion.md`
- 파일을 **종류별**(components/hooks/utils)로만 나눠 의존관계가 안 보임 → *함께 수정되는 파일을 같은 디렉토리에*
- 매직 넘버가 **함께 바뀌어야 할 값과 분리**되어 조용히 깨질 위험 → *매직 넘버 없애기*(응집도 관점)
- 폼 검증 로직의 **변경 단위**(필드별 vs 폼 전체)와 구조가 어긋남 → *폼의 응집도 생각하기*

### 결합도 → `references/coupling.md`
- 한 Hook/함수가 **너무 넓은 책임**을 떠안아 의존이 사방으로 퍼짐 → *책임을 하나씩 관리하기*
- **섣부른 공통화**로 인자가 비대해지고 수정 시 모든 사용처를 테스트해야 함 → *중복 코드 허용하기*
- 부모가 자식에게 prop을 **그대로 패스스루**(props drilling) → *Props Drilling 지우기* (Composition / Context)

## references 사용법

- 한 패턴을 정확히 적용·인용할 때는 해당 references 파일을 읽어 **원문 예제와 판단 기준**을 확인하세요. SKILL.md의 한 줄 요약만으로 코드를 고치지 마세요 — 근거(왜 그 개선이 변경 용이성을 높이는지)가 references에 있습니다.
- 각 파일은 4가지 기준 중 하나를 다루며, 패턴별로 `예시 / 냄새 / 개선 / 언제 적용·언제 하지 말지`를 담고 있습니다.
- 여러 기준이 동시에 걸리는 코드가 흔합니다(예: `usePageState`는 가독성+결합도, 매직 넘버는 가독성+응집도). 한 축만 보지 말고 교차 관점을 함께 검토하세요.

## 적용 범위와 태도

- 주로 React/TypeScript 프론트엔드를 전제하지만, 원칙(맥락 줄이기·예측 가능성·함께 바뀌는 것끼리 묶기·영향 범위 좁히기)은 일반 코드에도 적용됩니다.
- 출력 텍스트(설명·리뷰·주석)는 한국어로, 코드 식별자는 영어로 작성하세요.
- 이 스킬은 **나침반**이지 린터가 아닙니다. 모든 규칙을 강제하지 말고, 지금 코드가 장기적으로 변경하기 쉬워지는 방향을 같이 고민하세요.
